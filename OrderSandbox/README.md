# Тестовое задание — Вариант B: доработка шаблона

## Что это

Небольшой WPF-проект на .NET Framework 4.7.2, имитирующий модуль «Ручной заказ и дефектура»
из продакшен-проекта компании. Стилистически код написан близко к стилю основного проекта
(BaseViewModel с `RaisePropertyChanged`, `AsyncBase` для индикации загрузки, ICollectionView
для фильтрации), но не содержит реального бизнес-кода и данных.

Проект **уже открывается и запускается**, но в нём есть несколько намеренных проблем —
как обычных багов, так и недоделанной функциональности. Ваша задача — найти и
исправить их, а также реализовать одну небольшую фичу.

Это ближе к реальной работе в проекте, чем написание кода с нуля: вам нужно
сначала разобраться в существующем коде, понять задумку, а затем аккуратно
внести изменения, не поломав остальное.

## Как запустить

1. Откройте `OrderSandbox.sln` в Visual Studio 2019/2022 (с компонентом .NET desktop development).
2. Соберите и запустите проект (F5).
3. Откроется окно с двумя вкладками: «Ручной заказ» и «Дефектура».

Данные — тестовые, генерируются в памяти при старте (см. `Services/MockDataLoadingService.cs`),
никакая реальная БД не используется.

## Что нужно сделать

### 1. Кнопка «Добавить в заказ» неправильно блокируется

Сейчас кнопку можно нажать, даже если не выбран поставщик в списке предложений
(`SupplierOffers`). Это приводит к исключению NullReferenceException при попытке
добавить товар без выбранного предложения.

**Нужно**: сделать так, чтобы кнопка была недоступна, если:
- не выбран товар;
- не выбрано предложение поставщика;
- количество ≤ 0.

### 2. Нет проверки остатка и кратности упаковке

При добавлении в заказ количество может превышать `AvailableQuantity` у выбранного
поставщика — это нужно предотвратить или явно предупредить пользователя.

Также если `PackageSize > 1`, и введённое количество не кратно размеру упаковки,
пользователь должен получить понятное предупреждение (необязательно блокировать
добавление полностью — на ваше решение, опишите его в комментарии или в этом README).

### 3. Поиск не находит товары по нескольким словам

Поищите в списке товаров что-то вроде `"парац 500"` или `"анальгин 100"` — поиск
не сработает, хотя по требованиям должен находить товары, где встречаются ВСЕ
введённые фрагменты (в любом порядке, без учёта регистра), независимо от того,
встречаются они в коде товара или в названии.

Логика поиска находится в `Services/SearchService.cs`, метод `MatchesProduct`.

### 4. Ошибка загрузки данных проглатывается молча

В `Services/MockDataLoadingService.cs`, метод `LoadProductsAsync`, есть пустой
`catch`, который скрывает любую ошибку загрузки — пользователь просто увидит пустой
список товаров без всякого объяснения.

**Нужно**: убрать пустой catch и сделать так, чтобы в случае ошибки:
- пользователь увидел понятное сообщение об ошибке (например, через `ErrorMessage`
  во ViewModel, который уже есть и привязан в XAML, но не используется для этого случая);
- состояние загрузки (`AsyncBase`) корректно сбрасывалось, не оставляя UI в состоянии
  "вечной загрузки".

Чтобы проверить исправление, временно включите `ShouldSimulateFailure()` → `true`
в том же файле, убедитесь что ошибка отображается, затем верните `false`.

### 5. Удаление позиции из заказа не реализовано

Кнопка «Удалить» на вкладке «Ручной заказ» ничего не делает — метод
`RemoveFromOrderExecute()` в `ViewModels/OrderViewModel.cs` пустой.

**Нужно**: реализовать удаление выбранной строки из `OrderItems` и обновление
итоговой суммы заказа. Можно использовать `CommandParameter`, отдельное свойство
`SelectedOrderItem`, либо другой подход, который вам кажется более правильным —
выбор и обоснование оставляем на ваше усмотрение.

### 6. Фича: фильтр «Только в наличии»

Добавьте на вкладку «Ручной заказ» чекбокс или переключатель «Только товары с
остатком у хотя бы одного поставщика» — при включении список товаров должен
показывать только те товары, у которых есть хотя бы одно предложение поставщика
с `AvailableQuantity > 0`.

## Что не нужно делать

- Подключать реальную БД — все данные тестовые и генерируются в памяти.
- Реализовывать авторизацию, экспорт в Excel, и прочую функциональность за пределами
  описанных пунктов 1–6.
- Переписывать архитектуру целиком «для красоты» — задача про точечные правки в
  существующем коде, а не про рефакторинг с нуля.

## Что будет плюсом (не обязательно)

- Юнит-тесты для исправленного `SearchService`.
- Аккуратные комментарии о принятых решениях там, где условие задания допускает
  разные трактовки (например, как именно обрабатывать некратность упаковке).
- Сохранение существующего стиля кода (привязки через `RaisePropertyChanged(() => Prop)`,
  использование `AsyncBase` и т.д.) при внесении изменений.

## Критерии оценки

- **Корректность**: все 5 проблем исправлены, фича реализована и работает.
- **Стиль**: новый код вписывается в существующий стиль, а не вводит свой подход
  на каждый чих.
- **Валидация и обработка ошибок**: пользователь никогда не видит необработанное
  исключение, UI не виснет в состоянии загрузки.
- **MVVM**: минимум логики в code-behind, обращения к UI из ViewModel сведены
  к минимуму или отсутствуют.
- **Аккуратность изменений**: видно, что вы поняли существующий код перед тем,
  как его менять, а не переписали всё заново.
